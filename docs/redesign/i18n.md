# Internationalization (EN / HE / AR) + RTL

## Requirement (from client)

- Three languages: **English, Hebrew, Arabic.**
- Language switch **visible and easy to use, especially on mobile.**
- Hebrew and Arabic render **right-to-left (RTL).**
- **Do not rely on automatic translation.** Important pages must be checked by a real speaker.
- Everything localizes: **buttons, menus, forms, error messages, and the accessibility statement.**

## ⚠ Architecture decision — needs Faris's approval BEFORE coding

Do not pick an i18n stack unilaterally. First **inspect the repo**, report the framework/bundler and current routing, then propose one option below for approval:

- **If Vite + React (SPA):** `react-i18next` + `i18next`, locale in the URL path (`/`, `/he`, `/ar`), namespaced JSON per locale.
- **If Next.js (App Router):** `next-intl` or built-in i18n routing with locale segments (`/he/...`, `/ar/...`), server-rendered so language previews/SEO work.
- **Lightweight alternative:** compile-time messages (e.g. Paraglide) if bundle size matters.

Decision points to raise: **URL strategy** (path segment vs. subdomain vs. query — path is preferred for SEO + RTL routing), **default locale + fallback**, and **how translated content is stored** (JSON message catalogs vs. per-locale content files).

## RTL handling

- Set `dir="rtl"` and `lang` on `<html>` per active locale (`lang="he"` / `lang="ar"`, `dir="rtl"`).
- Prefer **logical CSS properties** (`margin-inline-start`, `padding-inline-end`, `inset-inline`) over left/right so layout mirrors automatically. Avoid hard-coded `left:`/`right:` in components.
- Mirror directional icons (arrows, chevrons); do **not** mirror the logo or photos.
- Numerals/dates: keep readable per locale; don't force Eastern-Arabic numerals unless the client wants them.
- Test the language switcher, mobile menu, and forms in all three directions.

## Translation workflow (important)

1. Claude Code extracts **all UI strings into an English source catalog** — no hard-coded text left in components.
2. HE and AR catalogs are **stubbed with keys + English fallback + a clear `// NEEDS TRANSLATION` marker** — Claude Code must **not** ship machine-translated Hebrew/Arabic as final copy.
3. Faris (trilingual) or a real speaker fills HE/AR. Approved copy replaces the stubs.
4. Verify no untranslated keys leak to production (missing-key handler logs, not silently blank).

**Scope note:** i18n is **Phase 3**. Land the restructured English homepage and accessibility first, then extract strings — retrofitting i18n onto a moving target wastes work. See the kickoff prompt's phasing.
